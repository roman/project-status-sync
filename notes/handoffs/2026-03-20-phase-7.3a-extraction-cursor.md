## Design Rationale

**Line-count cursor (not UUID-based)**: Follows approved proposal "incremental-extraction" (Option C).
- Simpler than UUID parsing (no `SessionEntry.uuid` field needed yet)
- Session JSONL files are append-only in practice (Claude Code only adds via `--continue`)
- Mirrors existing synthesis `Watermark` pattern (both use integer line counts)
- Falls back to `ExtractionCursor 0` (full transcript) on cursor miss

**Validation at parse boundary**: Smart constructor validates `n >= 0` in `readExtractionCursorFile`, not in the newtype constructor. This keeps the type simple while enforcing invariants.

**Separate cursor file** (`.extraction-cursors.json`):
- Not stored on `EventLogEntry` (avoids schema migration, `Maybe` pollution)
- Format: TBD by 7.3c (will be `Map SessionId ExtractionCursor` when wired into processSession)
