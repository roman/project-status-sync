# Test Plan: Phase 3.4 — Output Quality Validation

**WORKPLAN gate**: "Reading STATUS.md cold provides project understanding"
**Prerequisite**: Phase 2c.2 pipeline verification passed (outputs exist).
**Drives**: LLM agent reads this, walks human through assessment interactively.

---

## Agent Protocol

1. Work through steps in order.
2. These are subjective assessments — ask open-ended questions, not yes/no.
3. Record the human's verbatim assessment, then classify as PASS/FAIL.
4. If FAIL, ask follow-up questions to identify which prompt needs refinement.
5. Present results table and recommendations at the end.

---

## Prerequisites

Verify with the human:

- [ ] Phase 2c.2 pipeline passed — outputs exist in an output directory
- [ ] Ideally, some time has elapsed since the pipeline ran (hours, not minutes)
- [ ] Human is willing to read outputs without re-reading source transcripts first

If the human has run multiple sessions through the pipeline, use
accumulated outputs (richer data → better assessment).

---

## Step 1 — STATUS.md Cold Read

**Goal**: Can someone understand project state from STATUS.md alone?

Ask the human to open `STATUS.md` from the output directory without
looking at any other project docs first.

Then ask these questions **one at a time**, recording each answer:

1. "What is this project currently working on, based on STATUS.md?"
2. "What decisions have been made recently?"
3. "Are there any blockers or open questions mentioned?"
4. "If you were picking up this project after a week away, would this
   document orient you?"

**PASS**: Human answers at least 3 of 4 with substantive, accurate content.
**FAIL**: STATUS.md is too vague, too noisy, or misleading on 2+ questions.

**If FAIL**, ask: "What's missing or wrong? Is it too high-level, too
detailed, factually wrong, or just unhelpful?" — this guides synthesis
prompt revision.

---

## Step 2 — Handoff Usefulness

**Goal**: Handoff docs capture what a session actually accomplished.

Ask the human to:
1. Read one handoff file from the output.
2. Recall (or quickly skim) the original session.

Then ask:
- "Does this handoff accurately describe what happened in that session?"
- "Is anything important missing?"
- "Is anything there that shouldn't be (noise, hallucination)?"

Classify: **accurate** / **partially accurate** / **misleading**.

**PASS**: Accurate or partially accurate.
**FAIL**: Misleading or missing key information.

**If FAIL**, ask: "What specifically was wrong?" — this guides handoff
prompt revision.

---

## Step 3 — Progress Log Format

**Goal**: Progress entries are well-formatted and informative.

Ask the human to check `progress.log`:
- Does each line follow `YYYY-MM-DD HH:MM [prefix] — summary` format?
- Are the summaries meaningful (would you understand them in a month)?

**PASS**: Correct format and useful summaries.
**FAIL**: Malformed or cryptic entries.

---

## Step 4 — EVENTS.jsonl Signal Quality

**Goal**: Extracted events have good signal-to-noise ratio.

Ask the human to review `EVENTS.jsonl`:
- "Are most events things you'd actually want to know about later?"
- "Is there noise — events that don't matter?"
- "Is anything important from the session missing?"

Classify: **good signal** / **too noisy** / **too sparse** / **wrong focus**.

**PASS**: Good signal or minor issues.
**FAIL**: Too noisy, too sparse, or systematically wrong focus.

**If FAIL**, record which tags are over/under-represented — this guides
extraction prompt revision.

---

## Step 5 — Multi-Session Coherence (If Data Available)

**Goal**: Outputs stay coherent as sessions accumulate.

Skip this step if only one session has been processed.

If multiple sessions were processed, ask:
- "Does STATUS.md reflect the full arc, or just the latest session?"
- "Do handoff files feel like a coherent series?"
- "Does progress.log tell a story when read top to bottom?"

**PASS**: Outputs show accumulation, not just latest-session recency bias.
**FAIL**: Earlier sessions lost or dominated by latest.
